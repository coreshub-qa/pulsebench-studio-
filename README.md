# PulseBench Studio

PulseBench Studio 是一个面向 LLM 推理服务的单镜像压测实验台。你启动镜像后，直接在浏览器里完成压测配置、运行观察、结果分析与历史复盘，不需要手写 `evalscope perf` 命令。

## 项目简介

一句话概括：

**把 `evalscope perf` 这类强 CLI 能力，包装成一套适合日常实验、验证、汇报和复盘的可视化工作台。**

它适合以下场景：

- 新模型接入后，先做一次快速健康体检
- 准备上线前，验证不同输入输出长度与并发下的性能表现
- 做容量规划时，比较不同矩阵下的吞吐、延迟与成功率
- 在团队内沉淀标准测试模板，避免每次从零拼命令
- 利用 AI 帮助做结构化诊断和下一轮实验建议

## 背景与动机

LLM 推理压测通常并不缺底层工具，缺的是顺手的实验界面和可复用的工作流。

`evalscope perf` 本身已经很强，问题在于：

- 参数多，手动拼命令成本高
- 不同场景的测试矩阵难以重复组织
- 运行过程和原始日志对非作者不友好
- 报告虽然有数据，但不够适合快速决策
- 历史结果难以统一归档、检索与复跑
- 团队成员对“下一步该怎么测”经常没有共识

PulseBench Studio 解决的不是底层压测能力，而是**从配置、运行、解读到复盘的整条实验链路体验**。

## 解决什么问题

PulseBench Studio 主要解决以下问题：

### 1. 可视化配置替代手动拼命令

- 用表单配置模型地址、数据集、并发、请求数、输入输出长度
- 自动生成等效命令镜像，方便核对和复现
- 区分不同工作模式，避免所有用户都掉进同一张大表单

### 2. 实时监控压测过程

- 通过事件流展示运行阶段、日志与警告
- 在运行过程中明确当前状态，不需要盯后端终端
- 默认单活跃任务，减少多个压测互相污染结果

### 3. 结构化报告与图表展示

- 聚焦吞吐、延迟、TTFT、TPOT、ITL、成功率等指标
- 支持不同并发点的横向观察
- 把原始结果组织成更适合浏览的报告页

### 4. AI 诊断与下一轮建议

- 支持整体诊断、瓶颈判断、下一轮建议三种分析模式
- 可先测试 AI 模型配置是否可用，再开始分析
- AI 输出会做 Markdown 渲染与结果缓存，便于持续阅读和复盘

### 5. 历史归档与一键复测

- 自动保存单次 run 和批次 batch 的运行结果
- 支持历史检索、报告回看和快速复跑

## 核心功能

### 一键体检

用于首次摸底和快速验活。

特点：

- 只填模型连接信息即可启动
- 默认使用镜像内置轻量数据集
- 重点回答“服务现在是否健康、是否值得继续深入测”

### 场景模板

用于标准化验证和批量矩阵测试。

特点：

- 先选模板，再微调矩阵
- 适合沉淀团队标准实验方案
- 跑完直接进入批次报告

当前模板覆盖：

- 短文本体验
- 均衡吞吐
- 长上下文能力
- 容量压测

### 高级自定义

用于精细实验、复现实验和非标准验证。

特点：

- 保留完整参数控制权
- 连接信息、负载矩阵、输入输出边界分别配置
- 适合对单个实验做精细调参

### 实时事件流

- 以事件时间线展示运行过程
- 保留警告、日志和结束状态
- 适合在任务运行时快速判断是否异常

### 性能报告可视化

- 单次报告：查看本次 run 的概览、图表与诊断卡片
- 批次报告：对比不同场景或矩阵项的结果
- 原始摘要保留，便于追溯

### AI 分析工作区

- AI 连接测试：先验证 Base URL、API Key、模型是否真的可用
- 三种分析模式：
  - 整体诊断
  - 瓶颈判断
  - 下一轮建议
- 结果按 `run + 模式` 本地缓存，误关窗口后仍可继续查看
- 结果区支持 Markdown 渲染，适合阅读结构化分析

### 历史记录与重跑

- 展示 run 历史与 batch 历史
- 支持检索、回看报告和一键复跑

## 工作模式说明

首页会先让你选择入口，不同入口适合不同心智：

| 模式 | 适合场景 | 特点 |
| --- | --- | --- |
| 一键体检 | 首次接模型、快速摸底、排查服务是否可用 | 输入最少、最快出结论 |
| 场景模板 | 正式验证、团队复用、标准矩阵测试 | 目标清楚、批量运行、结果可比较 |
| 高级自定义 | 精细实验、非标准验证、复现实验 | 控制最完整、自由度最高 |

## 快速开始

### 方式一：Docker 部署（推荐）

推荐把模型目录挂载到 `/models`，数据集目录挂载到 `/datasets`，结果目录挂载到 `/data`。

```bash
docker build -t pulsebench-studio .

docker run -d \
  -p 8080:8080 \
  -v /host/pulsebench:/data \
  -v /host/models:/models \
  -v /host/datasets:/datasets \
  -e AI_BASE_URL=https://api.openai.com/v1 \
  -e AI_API_KEY=sk-xxx \
  -e AI_MODEL=gpt-4o-mini \
  pulsebench-studio
```

启动后访问：

```text
http://127.0.0.1:8080
```

### 方式二：Docker Compose 部署

```bash
docker compose up --build
```

默认 `docker-compose.yml` 会：

- 映射 `8080:8080`
- 挂载 `./data:/data`
- 挂载 `/tmp/models:/models`
- 挂载 `/tmp/datasets:/datasets`

如果你的模型与数据不在这两个目录，请按实际环境修改 `docker-compose.yml`。

### 方式三：本地开发

#### 后端

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app:app --reload --host 0.0.0.0 --port 8080
```

#### 前端

```bash
cd frontend
npm install
npm run dev
```

Vite 开发环境默认把 `/api` 代理到：

```text
http://127.0.0.1:8080
```

#### Makefile 快捷命令

```bash
make backend-dev
make frontend-dev
make build-frontend
```

## 环境变量说明

### AI 分析相关

| 变量名 | 说明 | 默认值 |
| --- | --- | --- |
| `AI_BASE_URL` | AI 服务 Base URL，要求兼容 OpenAI `chat/completions` | 空 |
| `AI_API_KEY` | AI 服务 API Key | 空 |
| `AI_MODEL` | 默认分析模型名称 | 空 |

说明：

- 页面里填写的 AI 配置会覆盖服务端环境变量
- 若页面里不填，则使用服务端环境变量
- 建议在分析前先点击“测试连接”

### 数据目录相关

| 变量名 | 说明 | 默认值 |
| --- | --- | --- |
| `PULSEBENCH_DATA_DIR` | 总数据目录 | `project_root/data` |
| `PULSEBENCH_RUNS_DIR` | 单次运行目录 | `${PULSEBENCH_DATA_DIR}/runs` |
| `PULSEBENCH_BATCHES_DIR` | 批次运行目录 | `${PULSEBENCH_DATA_DIR}/batches` |
| `PULSEBENCH_BUILTIN_DATASETS_DIR` | 内置数据集目录 | `backend/assets/datasets` |
| `PULSEBENCH_FRONTEND_DIST` | 前端构建产物目录 | `frontend/dist` |

### 调试相关

| 变量名 | 说明 | 默认值 |
| --- | --- | --- |
| `PULSEBENCH_ALLOW_MOCK` | 若本地缺少可执行依赖，可回退到 mock 数据验证链路 | `false` |

## 使用流程

### 1. 从首页选择入口

- 只想快速判断服务是否可用：选“一键体检”
- 想跑标准矩阵并批量对比：选“场景模板”
- 已经明确知道自己要测什么：选“高级自定义”

### 2. 配置测试参数

按页面填写：

- 模型名称
- API 地址
- API Key
- 并发列表
- 请求数列表
- 数据集模式
- Tokenizer / Dataset 路径
- 输入输出长度范围

注意：

- 页面里填的是**容器内路径**，不是宿主机路径
- 常见填写示例：
  - `/models/Qwen-1.5B`
  - `/datasets/prompts.txt`

### 3. 启动运行并监控过程

启动后会进入实时运行页，查看：

- 当前阶段
- 实时日志
- 警告信息
- 运行结束状态

默认策略：

- 同一时间只允许一个活跃任务
- 这样做是为了减少不同压测任务之间的结果污染

### 4. 查看报告与图表

完成后进入报告页，重点看：

- 总请求数
- 最佳 RPS
- 最低延迟
- 不同并发点的吞吐变化
- 百分位延迟
- 诊断卡片与原始摘要

### 5. 使用 AI 分析

在单次报告页点击 AI 分析工作区：

1. 配置 AI Base URL / API Key / Model
2. 点击“测试连接”确认模型可用
3. 选择分析模式
4. 可选填写附加问题
5. 点击开始分析

AI 分析适合回答的问题：

- 这次结果整体是否健康
- 当前瓶颈更像出在哪个阶段
- 下一轮最值得优先尝试什么参数

### 6. 回看历史与复跑

历史页会保存：

- 批次记录
- 单次运行记录

你可以：

- 搜索历史结果
- 回看单次或批次报告
- 基于历史 run 一键复跑

## 目录结构

```text
pulsebench-studio/
├── backend/                # FastAPI 后端、任务管理、报告归一化、AI 分析
├── frontend/               # React 前端工作台
├── data/                   # 默认数据目录
├── Dockerfile              # 单镜像构建
├── docker-compose.yml      # 本地 compose 启动
└── README.md
```

## 运行产物

每次单次 run 会写入：

```text
/data/runs/<run_id>/
├── spec.json
├── runtime.json
├── events.log
├── report.json
├── command.txt
└── raw/
```

批次相关结果会写入：

```text
/data/batches/<batch_id>/
```

## 适用对象

PulseBench Studio 适合：

- 推理服务工程师
- 模型平台工程师
- 需要做容量规划与基线验证的团队
- 希望把压测流程产品化、可视化的人

如果你只是想临时跑一条命令，CLI 就足够。

如果你想把压测变成**一套可复用、可观察、可汇报、可复盘的实验流程**，PulseBench Studio 更合适。

## 已知取舍

当前版本有这些明确取舍：

- 默认只允许一个活跃任务
- 不做数据库与多用户权限系统
- 历史页以检索和复跑为主，不做复杂实验管理
- AI 分析依赖外部兼容 OpenAI 的模型服务
- 核心目标是单机/单镜像实验台，不是大规模分布式压测平台
